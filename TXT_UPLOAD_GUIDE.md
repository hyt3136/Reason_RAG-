# 短对话TXT文件上传指南

## 📋 概述

本指南介绍如何上传和处理TXT格式的短对话文件，系统会自动进行：
1. 话题分段（识别对话中的话题转换）
2. 上下文增强（为每个话题添加前后文）
3. 自动标注（提取类型、角色、关键词等）
4. 向量化存储（存入RAG数据库）

## 📝 TXT文件格式要求

### 基本格式
- **每行一条对话**
- **支持编码**: UTF-8、GBK、GB2312
- **最小行长度**: 2个字符（更短的行会被过滤）

### 支持的格式

#### 格式1：纯文本（每行一条对话）
```
今天天气真好
是啊，要不要出去走走
好啊，去哪里
去公园吧
听说那边新开了一家咖啡店
那我们先去公园，然后去咖啡店
好的，几点出发
下午两点吧
```

#### 格式2：带说话人（推荐）
支持英文冒号 `:` 或中文冒号 `：`

```
元: 明年你高考要是能考到西农我给你发杯奶茶钱[旺柴]
菻荺: 我一定要考到西农吗？考其他的不可以[旺柴]
元: 不行
菻荺: [Emm]
```

或者：

```
小明：今天天气真好
小红：是啊，要不要出去走走
小明：好啊，去哪里
小红：去公园吧
```

**说话人格式特性**：
- ✅ 自动识别说话人
- ✅ 自动清理表情符号（如 `[旺柴]`、`[Emm]`）
- ✅ 保留说话人信息用于后续分析
- ✅ 支持中英文冒号混用

## 🚀 使用方法

### 方法1：交互模式（推荐新手）

```bash
python upload_txt_chat.py
```

然后按照提示：
1. 输入TXT文件路径
2. 选择是否自动标注（推荐选"是"）
3. 选择是否自动向量化（推荐选"是"）

### 方法2：命令行模式（推荐熟练用户）

#### 完整处理（话题分段 + 标注 + 向量化）
```bash
python upload_txt_chat.py --file "对话记录.txt"
```

#### 仅话题分段和上下文增强
```bash
python upload_txt_chat.py --file "对话记录.txt" --no-annotate
```

#### 指定输出目录
```bash
python upload_txt_chat.py --file "对话记录.txt" --output "./output"
```

#### 不自动向量化
```bash
python upload_txt_chat.py --file "对话记录.txt" --no-vectorize
```

## ⚙️ 配置参数

在 `config.py` 中可以调整以下参数：

### 话题分段配置
```python
TOPIC_SEGMENT_METHOD = "hybrid"  # 分段方法: vector/llm/hybrid
VECTOR_SIMILARITY_THRESHOLD = 0.7  # 相似度阈值
TIME_GAP_THRESHOLD = 300  # 时间间隔阈值（秒）
MIN_SEGMENT_LENGTH = 15  # 最小段落长度
MAX_SEGMENT_LENGTH = 200  # 最大段落长度
```

### 上下文窗口配置
```python
CONTEXT_WINDOW_SIZE = 2  # 前后各保留N个段落
MAX_CONTEXT_CHARS = 150  # 上下文最大字符数
INCLUDE_TOPIC_LABEL = True  # 是否包含话题标签
```

### TXT文件处理配置
```python
TXT_LINE_SEPARATOR = "\n"  # 行分隔符
TXT_MIN_LINE_LENGTH = 2  # 最小行长度
TXT_ENCODING_PRIORITY = ["utf-8", "gbk", "gb2312"]  # 编码尝试顺序
```

### 自动标注配置
```python
AUTO_ANNOTATE_BATCH_SIZE = 50  # 批量大小
AUTO_ANNOTATE_ENABLED = True  # 是否启用自动标注
```

## 🔍 处理流程详解

### 步骤1：话题分段
系统会自动识别对话中的话题转换点，将对话分成多个话题段落。

**分段策略**：
- **向量方法**: 基于语义相似度
- **LLM方法**: 使用大模型理解话题
- **混合方法**: 向量粗筛 + LLM精调（推荐）

### 步骤2：上下文增强
为每个话题段落添加前后文，解决短对话上下文依赖强的问题。

**增强策略**：
- 前文：保留前N个话题段落
- 后文：保留后N个话题段落
- 自动裁剪：控制上下文长度

### 步骤3：自动标注
使用LLM自动提取元数据：
- **类型**: 情感表达、问题讨论、日常闲聊、冲突争执
- **角色**: 对话参与者
- **关键词**: 核心话题词
- **摘要**: 20字以内总结
- **重要性**: high/middle/low

### 步骤4：向量化存储
将标注后的数据向量化并存入Chroma数据库，支持RAG检索。

## 💡 使用建议

### 1. 文件准备
- 确保每行是一条完整的对话
- 移除无意义的空行
- 统一编码格式（推荐UTF-8）

### 2. 参数调整
- **短对话**（<10字/条）：增大 `CONTEXT_WINDOW_SIZE`
- **长对话**（>50字/条）：减小 `MAX_SEGMENT_LENGTH`
- **话题跳跃频繁**：降低 `VECTOR_SIMILARITY_THRESHOLD`

### 3. 性能优化
- **大文件**（>1000行）：使用 `hybrid` 方法
- **小文件**（<500行）：使用 `llm` 方法
- **追求速度**：使用 `vector` 方法

### 4. 质量检查
处理完成后，建议检查：
- 话题分段是否合理
- 上下文是否完整
- 标注质量是否准确

## 🔧 故障排除

### 问题1：编码错误
**症状**: `UnicodeDecodeError`
**解决**: 
1. 检查文件编码
2. 调整 `TXT_ENCODING_PRIORITY` 顺序
3. 使用记事本另存为UTF-8格式

### 问题2：话题分段过细
**症状**: 生成了过多的小段落
**解决**: 
1. 增大 `MIN_SEGMENT_LENGTH`
2. 降低 `VECTOR_SIMILARITY_THRESHOLD`
3. 使用 `hybrid` 方法

### 问题3：话题分段过粗
**症状**: 不同话题被合并到一起
**解决**: 
1. 减小 `MAX_SEGMENT_LENGTH`
2. 提高 `VECTOR_SIMILARITY_THRESHOLD`
3. 使用 `llm` 方法

### 问题4：标注质量差
**症状**: 类型、关键词不准确
**解决**: 
1. 检查LLM配置
2. 减小 `AUTO_ANNOTATE_BATCH_SIZE`
3. 手动修正标注结果
